## Задание 1

**Опишите, какой тип резервного копирования и какую стратегию восстановления
стоит выбрать для каждого из сценариев.**

### Сценарий 1. Восстановить все данные за предыдущий день

Достаточно **полного (full) резервного копирования по расписанию раз в сутки**
(например, ночью, в период минимальной нагрузки). Восстановление — развёртывание
последнего суточного бэкапа.

- Плюсы: простота, один файл — одно состояние БД на момент снятия.
- Минусы: теряются изменения, сделанные после снятия бэкапа (потенциально до
  суток данных).
- Подходит, когда допустима потеря данных за время между бэкапами (RPO = сутки).

### Сценарий 2. Восстановить данные за час до сбоя

Нужна возможность восстановиться на произвольный момент времени — это
**Point-in-Time Recovery (PITR)**: полный бэкап + непрерывный архив журналов
(WAL в PostgreSQL, binlog в MySQL).

- Восстановление: разворачиваем последний полный бэкап и «накатываем» журналы
  до нужной отметки времени (`за час до сбоя`).
- Это даёт маленький RPO (минуты/секунды) ценой более сложной настройки и
  хранения журналов.
- Полное копирование раз в сутки тут уже недостаточно — оно не покрывает
  изменения внутри дня.

## Задание 2

**PostgreSQL: примеры команд резервного копирования и восстановления
(`pg_dump` / `pg_restore`) и возможности автоматизации.**

### Создание резервной копии

Логический бэкап одной базы в кастомном (сжатом) формате — рекомендуемый вариант
для `pg_restore`:

```bash
pg_dump -U postgres -h localhost -F c -f /backup/mydb.dump mydb
```

- `-F c` — custom-формат (сжатый, поддерживает выборочное восстановление);
- `-f` — файл назначения; `mydb` — имя базы.

Дамп в виде обычного SQL-скрипта:

```bash
pg_dump -U postgres -F p -f /backup/mydb.sql mydb
```

Бэкап всего кластера (все базы, роли, табличные пространства):

```bash
pg_dumpall -U postgres -f /backup/all.sql
```

### Восстановление

Из custom-формата (`pg_restore`), с предварительным созданием БД:

```bash
createdb -U postgres mydb_restored
pg_restore -U postgres -d mydb_restored /backup/mydb.dump
```

Из обычного SQL-дампа (через `psql`):

```bash
psql -U postgres -d mydb_restored -f /backup/mydb.sql
```

Полезные флаги `pg_restore`: `-j 4` — восстановление в 4 потока, `-c` —
предварительно удалить объекты, `-t table` — восстановить только одну таблицу.

## Задание 3

**MySQL: примеры команд инкрементального резервного копирования и когда
репликация выгоднее обычного бэкапа.**

### Полный логический бэкап (mysqldump)

```bash
mysqldump -u root -p --single-transaction --databases mydb > /backup/mydb.sql
```

Восстановление:

```bash
mysql -u root -p mydb < /backup/mydb.sql
```

### Инкрементальное резервное копирование

В MySQL инкрементальные бэкапы строятся на **бинарных логах (binlog)**: сначала
делается полный бэкап, затем сохраняются только последующие изменения из binlog.

Включение binlog (в `my.cnf`):

```conf
[mysqld]
server-id = 1
log_bin = /var/log/mysql/mysql-bin
```

Полный (базовый) бэкап с фиксацией позиции binlog:

```bash
mysqldump -u root -p --single-transaction --flush-logs --master-data=2 \
  --all-databases > /backup/full.sql
```

- `--flush-logs` — закрывает текущий binlog и начинает новый (точка отсчёта для
  инкремента);
- `--master-data=2` — записывает в дамп позицию binlog.

Инкремент — это просто новые binlog-файлы. Их можно копировать или
выгружать через `mysqlbinlog`:

```bash
# скопировать накопившиеся бинлоги
cp /var/log/mysql/mysql-bin.000123 /backup/inc/

# либо превратить их в SQL за нужный период
mysqlbinlog --start-datetime="2026-06-22 00:00:00" \
            --stop-datetime="2026-06-22 01:00:00" \
            /var/log/mysql/mysql-bin.000123 > /backup/inc/incr.sql
```

Восстановление = полный бэкап + накат бинлогов (в т.ч. PITR — до нужного момента):

```bash
mysql -u root -p < /backup/full.sql
mysqlbinlog /backup/inc/mysql-bin.000123 | mysql -u root -p
```

> Для физических инкрементальных бэкапов больших баз часто применяют
> **Percona XtraBackup** (`xtrabackup --backup --incremental-basedir=...`).
